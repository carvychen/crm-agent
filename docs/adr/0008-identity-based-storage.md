# Flex Consumption with identity-based storage (zero-secret by design)

Slice 11's delivery rehearsal (#24) proved that the original Bicep's **Y1 Linux Consumption** plan couldn't honour [ADR 0001](./0001-obo-with-wif.md)'s zero-long-lived-secrets story on a policy-locked landing zone (MCAPS-Hybrid, and almost certainly Lenovo's 21Vianet tenant). `AzureWebJobsStorage` was a shared-key connection string, the storage account had `allowSharedKeyAccess: false` forced by Azure Policy, and every modern deploy transport either required shared-key or was unavailable on Linux Consumption. In the middle of debugging we also noticed the Azure portal's explicit EOL notice: *"Linux Consumption will reach EOL on September 30 2028."*

The Function App now runs on **Flex Consumption (FC1)**. Flex natively supports identity-based deployment via `functionAppConfig.deployment.storage` + User-Assigned Managed Identity, and the runtime's `AzureWebJobsStorage` remains identity-based via the `__blobServiceUri / __queueServiceUri / __tableServiceUri / __credential=managedidentity / __clientId` pattern. The storage account has `allowSharedKeyAccess: false` explicitly in Bicep (defense-in-depth), the MI holds Storage Blob / Queue / Table Data Contributor at the storage-account scope, and the deployment container is declared in Bicep so the infrastructure contract is complete. Zero long-lived secret survives anywhere in the template.

## Considered Options

- **Retrofit identity-based storage on Y1 Linux Consumption** — *rejected after a half-built attempt.* Feasible in principle (the `AzureWebJobsStorage__*` quartet + role assignments + `WEBSITE_RUN_FROM_PACKAGE` via MI) but the supporting deploy transports on Y1 are stuck: `config-zip` uses shared-key; `az functionapp deploy --type zip` returns *"This API isn't available in this environment yet"*; Kudu `/api/zipdeploy` returns HTTP 400 with AAD tokens; `/api/publish?type=zip` returns 404; `WEBSITE_RUN_FROM_PACKAGE` with MI was set correctly but the Python worker never registered a function. On top of that, Linux Consumption EOLs 2028-09-30, so every hour spent working around Y1 limits is work we'd redo anyway.
- **Functions Premium (EP1+)** — rejected. Zero-secret story works, but Premium carries a minimum-1-instance cost floor that doesn't match our "100 calls/day" budget target; revisitable if scale justifies it later.
- **App Service Plan (B1+ or P0v3+)** — rejected. Same cost argument as Premium plus metrics-based scaling instead of event-driven; we gain nothing over Flex for this workload.
- **Container Apps Environment (ACA)** — rejected for this slice. Would change the programming model: `function_app.py` + `AsgiFunctionApp` stops being a Functions app and becomes a Dockerfile + explicit uvicorn + its own health/readiness contract. ADR 0002 (self-hosted MCP SDK on HTTP trigger) would need a new version. `scripts/preflight.py` would have to distinguish "Function App" from "ACA revision". That's 3× the scope and a different conversation; worth a separate proposal if the team later decides running-model flexibility outweighs the Functions SDK's convenience.
- **Keep shared-key and request per-customer policy exceptions** — rejected on Invariant 4 grounds. Every landing zone would need a governance exception, defeating delivered-blind.

## Consequences

- ADR 0001's zero-secret claim is now **architectural** rather than aspirational — no `storage.listKeys()`, no `AccountKey=` connection string, no shared-key in app settings. Any regression is a compile-time Bicep error.
- `az functionapp deployment source config-zip` is **back in use** — on Flex, the CLI automatically uploads the package to `functionAppConfig.deployment.storage`'s blob container using the site's Managed Identity. `bicep-deploy.md` §Zip-deploy is a simple one-liner again, with a note that identity-based is automatic on Flex.
- `host.json` becomes a committed file (it was previously missing; Slice 11 rehearsal surfaced that too).
- Linux Consumption's `FUNCTIONS_EXTENSION_VERSION`, `FUNCTIONS_WORKER_RUNTIME`, `linuxFxVersion`, and `WEBSITE_RUN_FROM_PACKAGE` app settings are all removed — Flex owns those via `functionAppConfig.runtime` and `functionAppConfig.deployment`.
- Function App code (`function_app.py`, `src/**`) and the Functions programming model are unchanged. `AsgiFunctionApp` runs identically on Flex.
- **Azure China (21Vianet) GA status must be verified for Lenovo**. Flex Consumption rolled into Azure China on a different timeline than Global. `docs/deployment/bicep-deploy.md` includes a pre-deploy check (`az functionapp list-flexconsumption-locations | grep -x <region>`) that fails fast if the target region is not GA; `docs/operations/troubleshooting.md` has the matching triage row. If Flex isn't GA in Lenovo's target region at deploy time, the next fallback is Functions Premium (least disruption) or Container Apps (larger but also zero-secret by design).
- **The azure-functions Python SDK ships a broken default route template for ASGI apps.** `AsgiFunctionApp` hardcodes `route="/{*route}"` with a leading slash; combined with any non-empty Functions route prefix, the resulting `<prefix>//{*route}` is rejected by the Flex host's ASP.NET Core 8 routing and kills JobHost startup. `src/flex_asgi.FlexAsgiFunctionApp` subclasses the SDK class and drops the leading slash; `host.json` sets `routePrefix=""` so the effective template is simply `{*route}`, matching what `src/asgi.py` mounts. A unit test (`tests/test_flex_asgi.py`) both pins our override and asserts the upstream bug still exists — once it's fixed upstream, the workaround can be deleted in a single PR.
- The three MI Data-role assignments + `allowSharedKeyAccess: false` would still be correct on Y1, Premium, or ACA — the security posture is portable even if the hosting model later changes.
- Y1 → FC1 is not an in-place Bicep update (different serverfarm sku family). Any RG deployed on Y1 must be torn down and recreated. The MI principal ID changes on recreate → the AAD app's FIC must be re-wired (aad-setup.md §3). Runbook documents this explicitly under "redeploy from a previous Y1 deployment".
